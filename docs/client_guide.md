# Руководство разработчика клиента

В данном руководстве описаны принципы создания клиентских приложений для Invisible Backend. Основной упор сделан на архитектуру клиента, реализацию сквозного шифрования (E2EE), работу с сетью и распространенные подводные камни.

> **Примечание:** Полная спецификация REST API и схемы JSON описаны в [api.md](./api.md).

---

## 1. Архитектура клиента

Клиентские приложения строятся на основе парадигмы **Zero Trust** (нулевого доверия). Это означает, что сервер рассматривается как потенциально скомпрометированный узел, который используется исключительно для маршрутизации зашифрованных данных и хранения публичных ключей.

### Основные принципы:
* **Локальное хранение:** Все приватные ключи генерируются и хранятся исключительно на устройстве клиента (рекомендуется использовать Secure Enclave / Keychain / Keystore).
* **Разделение каналов:** 
  * REST API используется для долговременных операций: аутентификация, публикация ключей, получение истории.
  * WebSocket используется для событий в реальном времени: новые сообщения, статусы присутствия, уведомления о доставке.
* **Local-First:** Клиент должен быть способен работать офлайн, сохраняя исходящие сообщения в локальную очередь для последующей отправки.

---

## 2. Сквозное шифрование (E2EE)

Взаимодействие между клиентами основывается на современных криптографических стандартах, аналогичных **протоколу Signal**.

### 2.1. X3DH (Extended Triple Diffie-Hellman)
Алгоритм X3DH используется для асинхронной установки защищенной сессии между двумя клиентами, даже если один из них находится в офлайне.

При регистрации или первом запуске на новом устройстве клиент генерирует и отправляет на сервер следующие публичные ключи (Bundle):
1. **Identity Key (IK):** Долгосрочный ключ идентификации (Ed25519 / Curve25519).
2. **Signed Pre-Key (SPK):** Среднесрочный ключ, подписанный Identity Key. Периодически обновляется (например, раз в неделю).
3. **One-Time Pre-Keys (OTPK):** Пул одноразовых ключей. При каждой инициализации сессии сервер выдает (и удаляет у себя) один OTPK.

Для начала переписки Алиса запрашивает Bundle Боба с сервера, выполняет 3 (или 4, если есть OTPK) операции Diffie-Hellman и вычисляет общий мастер-секрет (Shared Secret), который служит основой для инициализации сессии.

### 2.2. Double Ratchet
После вычисления мастер-секрета начинает работать алгоритм **Double Ratchet**. Он гарантирует два важнейших свойства безопасности:
* **Forward Secrecy (FS):** Компрометация текущих ключей не позволяет расшифровать прошлые сообщения.
* **Post-Compromise Security (PCS):** Если злоумышленник временно получил доступ к ключам, через несколько циклов обмена сообщениями сессия снова станет безопасной (самоисцеление).

Алгоритм состоит из двух "храповиков" (ratchet):
1. **KDF Ratchet (Симметричный):** Обновляет ключи шифрования для каждого нового сообщения (Message Key) с помощью Key Derivation Function (например, HKDF).
2. **DH Ratchet (Асимметричный):** При каждом ответе собеседника генерируется новая пара ключей Diffie-Hellman, и публичная часть прикрепляется к сообщению, обновляя корневой ключ (Root Key).

## 3. Рабочие процессы (Workflows)

### 3.1. Инициализация (Первый запуск)
1. Выполнение логина / регистрации через REST API.
2. Генерация IK, SPK и пула из ~100 OTPK.
3. Отправка публичных частей на сервер (см. эндпоинты ключей в `api.md`).
4. Сохранение приватных ключей в зашифрованное локальное хранилище.
5. Установка WebSocket-соединения с передачей JWT-токена.

### 3.2. Начало новой сессии
1. **Запрос ключей:** Клиент запрашивает с сервера набор ключей (Key Bundle) нужного пользователя.
2. **Вычисление X3DH:** Клиент вычисляет общий секрет.
3. **Первое сообщение:** Клиент шифрует первое сообщение с помощью инициализированного Ratchet-состояния. К этому сообщению **обязательно** прикрепляются параметры сессии (публичный IK, идентификатор использованного OTPK), чтобы получатель мог произвести аналогичные вычисления (Initial Message).

### 3.3. Прием первого сообщения
1. Сервер доставляет сообщение.
2. Получатель видит, что это Initial Message.
3. Достает из сообщения публичные ключи отправителя, свой задействованный OTPK (ищет его в локальном хранилище), вычисляет X3DH и инициализирует свой Ratchet.
4. Расшифровывает тело сообщения. Удаляет использованный OTPK из локального хранилища.

## 4. Офлайн-очереди (Offline Queuing)

Мобильные сети нестабильны, поэтому клиент должен быть устойчив к потере связи.

* **Локальная БД:** Все действия пользователя (отправка сообщений, прочтение) сначала записываются в локальную базу (например, SQLite, Realm, IndexedDB) со статусом `pending`.
* **Очередь отправки (Outbox):** Фоновый процесс следит за статусом сети. При появлении интернета он берет задачи из Outbox и пытается их выполнить.
* **Идемпотентность и Retry:** Отправка должна поддерживать повторные попытки (Exponential Backoff). Каждое сообщение имеет уникальный клиентский `message_id` (UUID), по которому сервер дедуплицирует запросы.
* **Порядок сообщений:** Клиент должен гарантировать, что сообщения уходят в правильном порядке, чтобы состояние Ratchet на стороне получателя не сломалось (хотя Double Ratchet поддерживает небольшие пропуски сообщений через временное хранение Skipped Message Keys).

## 5. Статус присутствия (Presence)

Функции Presence (онлайн, печатает) должны быть легковесными:
* **Трансляция статуса:** Клиент отправляет короткие WebSocket-сообщения типа `typing_start`, `typing_stop`.
* **Оптимизация:** Эти события **не шифруются по E2EE** (так как не содержат чувствительных данных), передаются только с авторизацией соединения и маршрутизируются сервером.
* **Игнорирование:** При нестабильной сети клиент может опускать отправку событий Presence, чтобы экономить трафик и батарею для критичных текстовых сообщений.
* **Таймауты:** Клиент должен сам снимать статус `typing`, если от собеседника не было обновлений более ~3-5 секунд.

## 6. Частые ошибки на клиенте (Client-Side Pitfalls)

При разработке E2EE-клиентов часто возникают специфические проблемы:

### 6.1. Исчерпание пула One-Time Pre-Keys (OTPK)
Сервер выдает OTPK другим пользователям для начала сессии с вами. Если они закончатся, fallback-механизм использует только SPK, что снижает безопасность (теряется гарантия безотказности прошлых сессий при компрометации SPK).
**Решение:** Клиент должен периодически запрашивать у сервера количество оставшихся OTPK и загружать новую партию, если их количество падает ниже критической отметки (например, < 20).

### 6.2. Потеря состояния Ratchet (Clear Data / Новое устройство)
Если пользователь удалил приложение или очистил данные, приватные ключи сессий будут потеряны.
**Решение:** При переустановке приложения клиент генерирует *новые* Identity Key и объявляет себя новым устройством (или заново инициирует сессии с контактами). Старые зашифрованные сообщения на сервере прочитать будет невозможно без восстановления бэкапа ключей.

### 6.3. Рассинхронизация сессий (Decryption Errors)
В редких случаях (например, из-за багов с очередностью сети или потери большого количества сообщений) состояние храповика у Алисы и Боба может разъехаться. Возникает постоянная ошибка расшифровки.
**Решение:** Клиент должен уметь ловить такие ошибки и отправлять специальное служебное сообщение "Session Reset" (Сброс сессии), чтобы прозрачно для пользователя заново выполнить процесс X3DH.

### 6.4. Уязвимости локального хранения
Ключи бессмысленны, если локальная база данных клиента не зашифрована.
**Решение:** Используйте встроенные средства ОС для шифрования данных в состоянии покоя: Keychain на iOS, Keystore на Android, WebCrypto API (с парольной деривацией ключа) в браузере.

### 6.5. Сохранение сообщений вне очереди
Состояния Ratchet-ключей детерминированы. Вызов функции шифрования изменяет состояние храповика (двигает цепь вперед).
**Ошибка:** Зашифровать сообщение, изменить стейт Ratchet, а затем *не сохранить* новое состояние в локальную БД из-за краша приложения. При следующем запуске вы зашифруете новое сообщение старым ключом, что сломает сессию.
**Решение:** Шифрование сообщения и сохранение обновленного состояния Ratchet в локальную базу данных должно происходить строго в рамках одной транзакции (ACID).
