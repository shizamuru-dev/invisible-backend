# Architecture Optimizations & E2EE Design

Текущая стратегия хранения и обмена сообщениями была значительно переработана с внедрением Redis для обеспечения высоких стандартов продакшена (HighLoad). Параллельно внедрена архитектура End-to-End Encryption.

---

## E2EE Architecture (End-to-End Encryption) - ✅ БАЗОВАЯ ИНФРАСТРУКТУРА ГОТОВА

### Принципы
Бэкенд работает как **слепой маршрутизатор (blind relay)** и **PKI-сервер (Key Distribution Server)**. Сервер никогда не видит открытый текст сообщений. Все криптографические операции выполняются исключительно на клиентах.

### Протоколы
| Тип чата | Протокол | Статус |
|----------|----------|--------|
| 1-1 (Direct Messages) | **Signal Protocol** (X3DH + Double Ratchet) | ✅ Инфраструктура готова |
| Группы / Каналы | **Olm/Megolm** (Matrix-style) | 🟡 Наработка (типы определены) |

### Ключевая схема (Signal Protocol)
Каждое устройство пользователя (= сессия в БД) загружает набор ключей:
- **Identity Key** — долгоживущий ключ идентификации устройства
- **Signed PreKey** — подписанный полу-эфемерный ключ (ротируется периодически)
- **One-Time PreKeys** — одноразовые ключи для инициации сессии (расходуемые)

### Таблицы БД
```
device_identity_keys   (user_username, device_id → identity_key, registration_id)
device_signed_pre_keys (user_username, device_id → key_id, public_key, signature)
device_one_time_keys   (user_username, device_id, key_id → public_key)  — расходуемые
message_ciphertexts    (message_id, device_id → ciphertext, signal_type) — per-device fanout
```

### API эндпоинты
- `POST /keys/upload` — устройство загружает свою связку ключей (Identity + Signed PreKey + OTKs)
- `GET /keys/claim/{username}` — клиент запрашивает ключи всех устройств собеседника для инициации X3DH. One-Time Key **атомарно потребляется** (UPDATE SET consumed=TRUE) при выдаче (через `FOR UPDATE SKIP LOCKED`). Возвращает `one_time_keys_remaining` для каждого устройства.
- `GET /keys/devices` — список всех устройств текущего пользователя (session + key metadata)
- `DELETE /keys/devices/{device_id}` — удаление устройства (cascade delete всех ключей). Владелец может удалить только своё устройство (403 для чужого).

### Поток зашифрованного сообщения
1. Клиент-отправитель шифрует сообщение для **каждого устройства** получателя (Device Fanout)
2. Отправляет WebSocket-сообщение `{"type": "Encrypted", "to": "bob", "id": "...", "ciphertexts": [...]}`
3. Relay маршрутизирует зашифрованный блоб через Redis Pub/Sub. Если получатель оффлайн — сохраняет в `offline_messages`.
4. Воркер сохраняет родительскую запись в `messages` (type=`encrypted`, content=NULL) и раскладывает `ciphertexts` в таблицу `message_ciphertexts`.

### `signal_type` значения
| Значение | Протокол | Описание |
|----------|----------|----------|
| 1 | Signal | Normal (Whisper) — сообщение в установленной сессии |
| 3 | Signal | PreKey — начальное сообщение (включает X3DH handshake) |
| 0 | Megolm (будущее) | Megolm ciphertext для групповых чатов |

### OTK consumed tracking
One-Time Keys больше не удаляются из БД при потреблении. Вместо этого `consumed` устанавливается в `TRUE`.partial index `(device_id) WHERE consumed = FALSE` обеспечивает эффективный поиск доступных ключей. Это позволяет вести аудит и историю использования OTK.

### claim_keys оптимизация (N+1 → 2 запроса)
Было: для каждого устройства 3 отдельных запроса (identity_key, signed_pre_key, pop OTK) → N×3 round-trips.
Стало: 2 запроса — JOIN для всех identity+signed_pre_key, batch DELETE+RETURNING для OTK (по одному на устройство). O(1) вместо O(N).

### Device Info при Login
При логине можно передать `device_info` (device_name, device_model, platform, hwid). Данные сохраняются в `sessions` и отображаются в `GET /keys/devices`.

### Ограничения текущей реализации
- Сервер не может выполнять поиск по контенту зашифрованных сообщений (by design).
- Push-уведомления не могут содержать текст сообщения; клиент должен расшифровывать локально.
- Rate limiting на `/keys/upload` отсутствует (TODO).

## 1. Блокировка и нагрузка на запись (High Write IOPS) - ✅ РЕШЕНО
**Проблема:** В MVP каждое отправленное сообщение заставляло микросервис `relay` делать синхронный `INSERT` в базу. Это генерировало огромный поток записей и блокировало event loop.
**Решение:** 
- **Асинхронные очереди (Redis Streams):** `relay` теперь только публикует сообщения в брокер (Redis Streams `XADD`).
- **Фоновый Batching (Воркер):** Отдельный фоновый процесс (пока запущен внутри `api`, но готов к выносу в отдельный микросервис) вычитывает поток (`XREADGROUP`), группирует события пачками по 100 штук за раз и делает `Bulk Insert` в PostgreSQL.
- **Pending Entries List (PEL):** Воркер при старте вычитывает непрочитанные/зависшие сообщения из своего PEL (`0`), гарантируя обработку At-Least-Once даже в случае падения во время транзакции в базу.
- **Раздельный ACK:** Непарсимые сообщения подтверждаются немедленно (чтобы не блокировать PEL), а успешно обработанные — только после коммита транзакции в БД.

## 2. Раздувание базы из-за UPDATE (MVCC Bloat) - 🟡 ТРЕБУЕТСЯ РЕШЕНИЕ
**Проблема:** При прочтении сообщения воркер делает батч-апдейт `UPDATE messages SET read_at = ... WHERE id IN (...)`. PostgreSQL использует механизм MVCC, где каждый `UPDATE` фактически создает новую строку на диске, а старая помечается как «мертвая» (до очистки процессом VACUUM). В чатах прочтение сообщений — это лавина апдейтов, что приводит к сильному раздуванию таблиц (table bloat) и деградации производительности.
**Решение:**
- Хранить статус прочтения не в строке каждого сообщения, а как отдельный "курсор" (Watermark) для диалога. Например, в отдельной таблице `dialog_states` с полями `(user_a, user_b, user_a_last_read_msg_id, user_b_last_read_msg_id)`. Тогда вместо обновления 100 сообщений обновляется только 1 строка с ID последнего прочитанного сообщения.

## 3. Бесконечный рост таблицы (Table Size) - 🟡 ТРЕБУЕТСЯ РЕШЕНИЕ
**Проблема:** Все сообщения складываются в одну таблицу `messages`. Со временем таблица вырастет до сотен миллионов строк, выборка начнет деградировать, а обслуживание БД (бэкапы, VACUUM, миграции) станет мучительным.
**Решение:**
- **Партицирование (Partitioning):** Разбить таблицу `messages` в PostgreSQL по времени (например, по месяцам или неделям).
- При гигантских масштабах (уровня Discord, Telegram) реляционные БД для истории сообщений обычно заменяют на NoSQL решения, оптимизированные для колоссального потока записей и чтений по ключу (ScyllaDB, Cassandra, DynamoDB).

## 4. Кэширование сессий и сообщений - ✅ СЕССИИ ГОТОВЫ
**Проблема:** Изначально валидация токенов и загрузка сообщений создавали нагрузку на PostgreSQL.
**Текущее решение:** Сессии теперь кэшируются в Redis (ключ `session:{id}`) при логине, обновляют TTL при активности и удаляются при `/api/auth/logout`.
**Оставшаяся проблема:** API `GET /api/messages/{username}` все еще идет напрямую в PostgreSQL. При открытии приложения пользователи запрашивают историю последних диалогов, создавая пиковую нагрузку на чтение.
**Решение:**
- Кэшировать последние 50-100 сообщений активных диалогов в Redis. При запросе API сначала отдает данные из Redis, и только при скролле вверх (пагинации) идет в PostgreSQL за старыми сообщениями.

## 5. Database Migrations - ✅ РЕШЕНО
**Проблема:** `api` микросервис создавал таблицу `users` вручную через `CREATE TABLE IF NOT EXISTS` в `main()`, не запуская полный набор миграций. Это приводило к тому, что E2EE таблицы и другие таблицы из `migrations/` не существовали при свежем старте `api`.
**Решение:** Оба микросервиса (`api` и `relay`) теперь запускают `sqlx::migrate!("../migrations")` при старте. Inline `CREATE TABLE` удалён.

## 6. Client-Side File Encryption - 🟡 В ПРОЦЕССЕ
Файлы в текущей реализации uploadятся в S3/MinIO в plaintext (только presigned URL как временный токен). Для полноценной E2EE-системы файлы должны шифроваться на клиенте перед загрузкой.
**Текущее решение (на клиенте):** Клиент генерирует случайный AES-256 ключ, шифрует файл, загружает зашифрованный blob в S3. Ключ файла передаётся получателю через Signal Protocol encrypted message payload (часть Olm/Megolm ratchet).
