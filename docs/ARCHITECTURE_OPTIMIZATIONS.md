# Future Architecture Optimizations (Production Readiness)

Текущая стратегия хранения и обмена сообщениями была значительно переработана с внедрением Redis для обеспечения высоких стандартов продакшена (HighLoad). Однако остаются зоны для дальнейшего улучшения.

## 1. Блокировка и нагрузка на запись (High Write IOPS) - ✅ РЕШЕНО
**Проблема:** В MVP каждое отправленное сообщение заставляло микросервис `relay` делать синхронный `INSERT` в базу. Это генерировало огромный поток записей и блокировало event loop.
**Решение:** 
- **Асинхронные очереди (Redis Streams):** `relay` теперь только публикует сообщения в брокер (Redis Streams `XADD`).
- **Фоновый Batching (Воркер):** Отдельный фоновый процесс (пока запущен внутри `api`, но готов к выносу в отдельный микросервис) вычитывает поток (`XREADGROUP`), группирует события пачками по 100 штук за раз и делает `Bulk Insert` в PostgreSQL.
- **Pending Entries List (PEL):** Воркер при старте вычитывает непрочитанные/зависшие сообщения из своего PEL (`0`), гарантируя обработку At-Least-Once даже в случае падения во время транзакции в базу.

## 2. Раздувание базы из-за UPDATE (MVCC Bloat) - 🟡 ТРЕБУЕТСЯ РЕШЕНИЕ
**Проблема:** При прочтении сообщения воркер делает батч-апдейт `UPDATE messages SET read_at = ... WHERE id IN (...)`. PostgreSQL использует механизм MVCC, где каждый `UPDATE` фактически создает новую строку на диске, а старая помечается как «мертвая» (до очистки процессом VACUUM). В чатах прочтение сообщений — это лавина апдейтов, что приводит к сильному раздуванию таблиц (table bloat) и деградации производительности.
**Решение:**
- Хранить статус прочтения не в строке каждого сообщения, а как отдельный "курсор" (Watermark) для диалога. Например, в отдельной таблице `dialog_states` с полями `(user_a, user_b, user_a_last_read_msg_id, user_b_last_read_msg_id)`. Тогда вместо обновления 100 сообщений обновляется только 1 строка с ID последнего прочитанного сообщения.

## 3. Бесконечный рост таблицы (Table Size) - 🟡 ТРЕБУЕТСЯ РЕШЕНИЕ
**Проблема:** Все сообщения складываются в одну таблицу `messages`. Со временем таблица вырастет до сотен миллионов строк, выборка начнет деградировать, а обслуживание БД (бэкапы, VACUUM, миграции) станет мучительным.
**Решение:**
- **Партицирование (Partitioning):** Разбить таблицу `messages` в PostgreSQL по времени (например, по месяцам или неделям).
- При гигантских масштабах (уровня Discord, Telegram) реляционные БД для истории сообщений обычно заменяют на NoSQL решения, оптимизированные для колоссального потока записей и чтений по ключу (ScyllaDB, Cassandra, DynamoDB).

## 4. Кэширование сессий и сообщений - ✅ СЕССИИ ГОТОВЫ
**Проблема:** Изначально валидация токенов и загрузка сообщений создавали нагрузку на PostgreSQL.
**Текущее решение:** Сессии теперь кэшируются в Redis (ключ `session:{id}`) при логине, обновляют TTL при активности и удаляются при `/api/auth/logout`.
**Оставшаяся проблема:** API `GET /api/messages/{username}` все еще идет напрямую в PostgreSQL. При открытии приложения пользователи запрашивают историю последних диалогов, создавая пиковую нагрузку на чтение.
**Решение:**
- Кэшировать последние 50-100 сообщений активных диалогов в Redis. При запросе API сначала отдает данные из Redis, и только при скролле вверх (пагинации) идет в PostgreSQL за старыми сообщениями.
